# 성능 / 트래픽 테스트 계획

## 문서 목적

- 대규모 요청이 들어오는 상황에서 무엇을 먼저 검증해야 하는지 정리한다.
- 어떤 도구로 어떤 방식으로 테스트를 수행할지 기준을 만든다.
- 초기 트래픽 방어 테스트와 이후 고도화 테스트를 구분해 단계적으로 진행할 수 있게 한다.
- 성능 테스트 결과를 해석할 때 봐야 할 핵심 지표와 성공 조건을 정리한다.

## 전제 조건

이 문서는 `모든 일반 기능이 의도한 대로 동작한다`는 전제 위에서 시작한다.

즉, 성능 테스트를 본격적으로 시작하기 전에 아래 조건을 먼저 만족시키는 것을 기준으로 한다.

- 인증, 조회, 홀드, 예약, 결제, 티켓, 체크인, 관리자 API의 정상 기능이 확인되었다.
- 주요 실패 케이스와 권한 검증이 확인되었다.
- 통합 테스트 또는 수동 기능 검증으로 핵심 흐름이 깨지지 않음을 확인했다.

일반 기능 시나리오는 [FUNCTIONAL_TEST_SCENARIOS.md](FUNCTIONAL_TEST_SCENARIOS.md)를 참고한다.

## 기본 방향

이 프로젝트의 성능 테스트는 단순 부하 수치 경쟁보다 다음 두 가지를 우선 목표로 삼는다.

- 예매 정합성 보호
  같은 좌석 중복 홀드, 초과 판매, 중복 결제, 중복 체크인이 없어야 한다.
- 오픈 직후 트래픽 방어
  회차 조회, 로그인, 홀드 생성, 예약 전환, 결제 승인 구간이 급격히 몰려도 시스템이 예측 가능한 방식으로 버텨야 한다.
- 1차 범위는 백엔드 단독 검증
  프론트엔드와 nginx를 통하지 않고, 백엔드 API를 직접 호출해 애플리케이션과 DB가 어디까지 감당 가능한지 먼저 확인한다.

즉, 이 문서의 테스트는 `빠른가`만 보는 것이 아니라 `안전하게 동작하는가`를 함께 본다.

## 권장 도구

### 1. `JMeter` 와 `k6` 비교

두 도구 모두 백엔드 부하 테스트에 사용할 수 있다.
다만 이 프로젝트의 1차 목적은 `백엔드 API를 빠르게 반복 검증하고, 시나리오를 저장소에 유지하는 것`이기 때문에 선택 기준을 분명히 둘 필요가 있다.

| 비교 항목 | JMeter | k6 | 이 프로젝트 기준 판단 |
| --- | --- | --- | --- |
| 시작 방식 | GUI로 테스트 플랜을 만들고, 실제 부하는 CLI로 실행 | JavaScript/TypeScript 스크립트를 작성하고 바로 CLI 실행 | 개발자가 반복 실행하기에는 `k6`가 더 가볍다 |
| 시나리오 관리 | `jmx` 기반이라 구조가 커지면 diff와 리뷰가 불편해질 수 있음 | 텍스트 스크립트라 git diff, 코드 리뷰, 재사용이 쉽다 | 저장소 중심 운영에는 `k6`가 유리하다 |
| 인증/토큰/상태 전이 | 가능하지만 요소가 많아질수록 화면 구성이 복잡해질 수 있음 | 코드로 분기와 토큰 재사용을 표현하기 쉬움 | 예매 흐름 API에는 `k6`가 더 자연스럽다 |
| 부하 프로파일 구성 | Thread Group 기반으로 충분히 구성 가능 | `stages`, `scenarios`, arrival-rate 모델 등으로 세밀하게 구성 가능 | 오픈 직후 트래픽 모델링에는 `k6`가 편하다 |
| 합격/실패 기준 자동화 | 결과 파일과 HTML 리포트 분석에 강점 | `threshold`로 pass/fail 기준을 테스트 코드에 직접 남길 수 있음 | 자동 게이트는 `k6`가 더 명확하다 |
| 입문 편의성 | GUI가 있어 비개발자도 접근하기 쉽다 | 스크립트 작성이 필요하지만 HTTP API 테스트는 빠르게 적응 가능 | 팀이 코드 중심이면 `k6`, GUI 선호면 `JMeter` |
| 결과 확인 | HTML 대시보드 생성이 강점 | 콘솔 요약과 메트릭 기준 판별이 직관적 | 1차 검증은 둘 다 가능하지만 자동 반복은 `k6`가 단순하다 |

### 2. 1차 선택: `k6`

이 프로젝트의 1차 성능 테스트 도구는 `k6`를 권장한다.

선정 이유:

- 현재 테스트 대상이 `브라우저 사용자 경험`이 아니라 `백엔드 HTTP API + PostgreSQL + Redis` 조합이기 때문이다.
- 인증, 조회, 홀드, 예약, 결제처럼 상태 전이와 토큰 흐름이 섞인 API 시나리오를 코드로 관리하는 편이 훨씬 읽기 쉽고 유지보수하기 쉽다.
- 테스트 스크립트를 저장소에 같이 두고 diff, 리뷰, 재실행하기에 `k6`가 더 적합하다.
- 응답 시간, 에러율, 체크 결과를 `threshold`로 남겨서 `이번 실행이 통과인지 실패인지`를 명확히 판단하기 좋다.
- 이후 CI나 스테이징 검증으로 확장할 때도 같은 스크립트를 그대로 재사용하기 쉽다.

즉, `JMeter가 안 좋아서`가 아니라, `이번 1차 목표가 backend-only API 부하 검증이고 팀 운영 방식이 문서/코드 중심`이기 때문에 `k6`가 더 잘 맞는다.

### 3. `JMeter` 를 선택하는 편이 더 나은 경우

아래 상황이라면 `JMeter`를 우선 선택해도 된다.

- GUI로 테스트 플랜을 눈으로 조립하고 싶은 경우
- 비개발자도 직접 시나리오를 수정해야 하는 경우
- 브라우저/프록시 기록 기반으로 빠르게 샘플 테스트 플랜을 만들고 싶은 경우
- HTML 리포트를 중심으로 결과 공유를 먼저 하고 싶은 경우

### 4. 현재 프로젝트에서의 결론

현재 백엔드 1차 부하 테스트는 다음 기준으로 진행한다.

- 대상:
  프론트엔드, nginx를 제외한 `backend API 직접 호출`
- 1차 도구:
  `k6`
- JMeter 위치:
  보조 대안 또는 GUI 중심 테스트 플랜이 필요할 때 사용

권장 사용 범위:

- 공개 조회 API 읽기 폭주 테스트
- 로그인 / 토큰 재발급 부하 테스트
- 홀드 / 예약 / 결제 API 동시 요청 테스트
- 혼합 시나리오 테스트

### 5. 대안 도구

- `Gatling`
  JVM 친화적이고 시나리오 구성이 강력하다.
- `JMeter`
  GUI 중심 접근이 필요할 때 유효한 대안이다.
- `wrk`, `hey`
  단순 읽기 API에 대한 짧은 벤치마크에는 유용하지만, 인증과 상태 전이가 섞인 복합 시나리오에는 한계가 있다.

## 권장 관측 도구

현재 코드 기준으로는 본격적인 메트릭 수집 체계가 아직 충분하지 않다.
그래서 성능 테스트를 시작할 때는 아래 관측 수단을 함께 준비하는 것이 좋다.

- 애플리케이션 로그
- DB 로그
- `docker stats` 또는 OS 수준 CPU / 메모리 관측
- PostgreSQL slow query / lock wait 확인
- 추후 도입 권장:
  `Actuator + Micrometer + Prometheus + Grafana`

## 권장 실행 환경

### 1단계: 로컬 검증

- 목적:
  시나리오와 데이터 준비, 기본 동작, 실패 조건을 빠르게 검증한다.
- 환경:
  개발자 PC + 로컬 PostgreSQL/Redis 또는 Docker Compose
- 특징:
  절대 수치보다 시나리오 유효성 검증이 목적이다.

### 2단계: Docker Compose 검증

- 목적:
  컨테이너 기반 실행에서 초기 병목과 설정 문제를 확인한다.
- 환경:
  workspace root `docker compose up --build`
- 특징:
  nginx 프록시, backend, postgres, redis를 함께 본다.

### 3단계: 스테이징 유사 환경

- 목적:
  실제 운영 전제에 가까운 인프라에서 부하를 본다.
- 환경:
  별도 서버 또는 클라우드 환경, 외부 로드 밸런서, 실제와 유사한 DB 크기
- 특징:
  여기서부터는 성능 수치와 안정성 판단에 의미를 두기 시작한다.

## 테스트 전 준비 사항

성능 테스트는 사전 데이터와 측정 기준이 없으면 결과 해석이 어렵다.
테스트 전에 아래를 준비한다.

### 데이터 준비

- 테스트용 이벤트 / 회차 / 좌석 / 구역 생성
- 좌석형, 구역형 인벤토리 모두 포함
- 로그인용 일반 사용자 여러 명 준비
- 관리자 계정 준비

### 기준값 준비

- 목표 응답 시간
- 허용 가능한 에러율
- 허용 가능한 DB CPU / lock wait 수준
- 테스트 중 절대 깨지면 안 되는 정합성 규칙

### 환경 준비

- 로그 수집 위치 확인
- DB 접속 정보와 모니터링 수단 확인
- 테스트 종료 후 데이터 정리 방법 확인

## 추천 폴더 구조

테스트 스크립트를 추가할 때는 아래 구조를 권장한다.

```text
backend/
  perf/

```

예시:

- `perf/k6/load/show-availability.js`
- `perf/k6/load/hold-seat-race.js`
- `perf/k6/load/payment-idempotency.js`

## `k6` 기준 진행 방법

### 1. 환경 준비

백엔드를 먼저 실행한다.

로컬 예시:

```powershell
./gradlew.bat bootRun
```

또는 Compose 예시:

```powershell
docker compose up --build
```

### 2. 테스트 데이터 준비

- 관리자 API로 공연장, 이벤트, 회차, 좌석, 구역, 회차 인벤토리를 생성한다.
- 또는 로컬 샘플 데이터를 활용한다.

### 3. 인증 토큰 준비

- 사용자 예매 API 테스트에는 로그인 후 Access Token이 필요하다.
- 관리자 API 기반 데이터 준비에는 관리자 토큰이 필요할 수 있다.

### 4. 시나리오 실행

예시 명령:

```powershell
k6 run .\perf\k6\load\show-availability.js
```

환경변수를 함께 주는 예시:

```powershell
$env:BASE_URL="http://localhost:9090"
$env:ACCESS_TOKEN="..."
k6 run .\perf\k6\load\hold-seat-race.js
```

### 5. 결과 기록

매 실행마다 아래 항목을 남긴다.

- 실행 일시
- 브랜치 / 커밋 SHA
- 테스트 환경
- 시나리오 이름
- 가상 사용자 수 / 도착률 / 실행 시간
- 평균 / `p95` / `p99`
- 에러율
- 정합성 검증 결과

## 1차 핵심 시나리오

아래 시나리오는 우선순위가 높은 순서대로 정리했다.

### PERF-001 공개 조회 폭주

- 대상 API:
  `GET /api/shows/{showId}/availability`
- 목적:
  예매 오픈 직후 가장 먼저 몰릴 읽기 트래픽에 대한 병목 확인
- 방식:
  같은 `showId`에 대해 다수의 동시 조회 요청 수행
- 확인 포인트:
  응답 시간, 에러율, DB CPU, 정렬 비용, 반복 조회 비용
- 성공 조건:
  읽기 폭주 상황에서도 5xx가 거의 없어야 하고, 응답 시간이 급격히 무너지지 않아야 한다.

### PERF-002 이벤트 / 회차 조회 혼합 읽기

- 대상 API:
  `GET /api/events`
  `GET /api/events/{eventId}`
  `GET /api/shows/{showId}/availability`
- 목적:
  사용자가 실제 브라우징하는 흐름에서 공개 조회 API 조합의 부담 확인
- 방식:
  목록, 상세, 가용성 조회를 비율로 섞어 호출
- 확인 포인트:
  특정 엔드포인트 쏠림 여부, 평균 응답 시간, 캐시 필요성

### PERF-003 동일 좌석 홀드 경쟁

- 대상 API:
  `POST /api/holds`
- 목적:
  같은 좌석에 대한 동시 선점 경쟁에서 정합성 보장 확인
- 방식:
  여러 사용자로 같은 `seatId` 홀드 요청 동시 전송
- 확인 포인트:
  성공 건수, 실패 응답 형태, 좌석 상태 최종값
- 성공 조건:
  정확히 한 요청만 성공하고 나머지는 중복 점유 없이 실패해야 한다.

### PERF-004 동일 구역 수량 경쟁

- 대상 API:
  `POST /api/holds`
- 목적:
  구역형 재고에서 초과 판매가 없는지 확인
- 방식:
  남은 수량보다 큰 총합이 되도록 동시에 홀드 요청 전송
- 확인 포인트:
  `hold_qty`, `sold_qty`, 남은 수량, 실패율
- 성공 조건:
  capacity를 넘는 점유가 발생하면 안 된다.

### PERF-005 예약 전환 경쟁

- 대상 API:
  `POST /api/reservations`
- 목적:
  같은 hold 또는 같은 자원에 대한 예약 전환 충돌 처리 확인
- 방식:
  동일 hold를 기준으로 예약 생성 경쟁 유도
- 확인 포인트:
  중복 예약 생성 여부, hold 상태, 좌석 상태, reservation 개수
- 성공 조건:
  하나의 hold가 중복 예약으로 전환되면 안 된다.

### PERF-006 결제 멱등성 검증

- 대상 API:
  `POST /api/payments/confirm`
- 목적:
  같은 결제 요청의 재시도와 중복 요청 처리 확인
- 방식:
  같은 `Idempotency-Key`로 같은 요청 재전송
- 추가 케이스:
  같은 키로 다른 금액, 다른 reservationId 요청도 전송
- 확인 포인트:
  payment 개수, ticket 개수, reservation 상태
- 성공 조건:
  같은 요청은 같은 결과를 반환해야 하고, 중복 티켓 발급이 없어야 한다.

### PERF-007 인증 폭주

- 대상 API:
  `POST /api/auth/login`
  `POST /api/auth/refresh`
- 목적:
  예매 시작 직전/직후 로그인과 재발급 집중 상황 확인
- 방식:
  다수 사용자 계정으로 로그인 / 재발급 요청 반복
- 확인 포인트:
  인증 응답 시간, DB 조회 비용, 토큰 저장 비용, 실패율

### PERF-008 혼합 시나리오

- 대상 API:
  공개 조회 + 로그인 + 홀드 + 예약 + 결제
- 목적:
  실제 예매 흐름에 가까운 종합 부하 확인
- 권장 비율 예시:
  조회 70, 로그인/재발급 10, 홀드 10, 예약 5, 결제 5
- 확인 포인트:
  특정 API만 볼 때와 달리 전체 흐름에서 병목이 어디로 이동하는지 확인

### PERF-009 Soak Test

- 목적:
  짧은 순간 최대치보다, 일정 시간 동안 시스템이 안정적으로 유지되는지 확인
- 권장 시간:
  최소 30분, 가능하면 1시간 이상
- 확인 포인트:
  메모리 증가 추세, 커넥션 누수, 응답 시간 점진적 악화 여부

### PERF-010 장애 유도 / 복구 시나리오

- 목적:
  장애 징후가 들어왔을 때 시스템이 어떻게 반응하는지 확인
- 예시:
  DB 지연, DB 커넥션 부족, backend graceful shutdown 중 요청 유입
- 확인 포인트:
  실패 응답 일관성, 재시도 필요 여부, 장애 후 회복 시간

처음부터 모든 시나리오를 크게 돌리기보다 아래 순서를 권장한다.

2. `PERF-003` 동일 좌석 홀드 경쟁
3. `PERF-004` 동일 구역 수량 경쟁

이 순서로 가면 읽기 병목과 정합성 이슈를 먼저 확인하고, 이후 종합 흐름과 장시간 안정성으로 넘어갈 수 있다.

## 결과 해석 기준

성능 테스트 결과를 볼 때 아래 질문에 답할 수 있어야 한다.

- 어떤 API가 가장 먼저 무너지는가
- 성능보다 먼저 정합성이 깨지는가
- DB lock wait가 급증하는 지점은 어디인가
- CPU, 메모리, 커넥션 중 먼저 바닥나는 자원은 무엇인가
- 읽기 병목인지, 쓰기 경합인지, 인증 병목인지 구분되는가

## 현재 코드 기준 특별히 볼 부분

현재 구현 기준으로 특히 주의해서 봐야 할 항목은 아래와 같다.

- 회차 가용성 조회는 읽기 폭주 시 첫 번째 병목 후보다.
- 홀드 / 예약 / 결제는 정합성 보호를 위해 락이 걸리는 구간이므로 lock wait를 꼭 봐야 한다.
- 결제 멱등성은 정상 재시도뿐 아니라 경쟁 상황에서도 같은 결과를 유지하는지 확인해야 한다.
- Hold / Reservation 만료 해제는 lazy release 중심이므로 만료 자원이 많은 상황에서 조회/생성 API에 어떤 추가 비용이 붙는지 봐야 한다.
