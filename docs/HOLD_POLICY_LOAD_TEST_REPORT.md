# Hold 좌석 선점 정책 부하 테스트 리포트

작성일: 2026-05-20

## 1. 목적

한 좌석에 완전동시 요청이 몰리는 티켓팅 상황에서 좌석 선점 정책별 정합성, 응답 속도, 장애 가능성을 비교했다.

비교 대상은 다음 네 가지이다.

| 구분 | 브랜치 | 설명 |
| --- | --- | --- |
| main | `main` | 기존 DB lock 기반 방식 |
| 좌석별 큐 제한 | `codex/seat-queue-limit` | main 기준으로 좌석별 동시 진입 수 제한 추가 |
| Redis pre-lock | `codex/redis-seat-prelock` | DB 진입 전에 Redis로 좌석 선점 충돌을 빠르게 차단 |
| 좌석별 큐 제한 + Redis pre-lock | `codex/redis-prelock-seat-queue-limit` | 좌석별 큐 제한과 Redis pre-lock을 함께 적용 |

## 3. 구현 요약

### 3.1 좌석별 큐 제한

좌석별 큐 제한은 `CreateHoldUseCase` 진입부에서 먼저 수행된다. 요청한 좌석에 대해 Redis set 기반 큐 티켓을 획득하지 못하면 DB까지 진입하지 않고 빠르게 실패한다.

주요 구성은 다음과 같다.

| 파일 | 역할 |
| --- | --- |
| `hold/application/SeatHoldQueueManager.java` | 좌석별 큐 진입/퇴장 인터페이스 |
| `hold/application/SeatHoldQueueTicket.java` | 큐 점유 티켓 |
| `hold/infrastructure/redis/RedisSeatHoldQueueManager.java` | Redis 기반 큐 제한 구현 |
| `hold/infrastructure/memory/NoOpSeatHoldQueueManager.java` | 큐 제한 비활성화 시 사용하는 대체 구현 |
| `hold/application/CreateHoldUseCase.java` | hold 생성 전에 좌석 큐 진입 수행 |

기본 설정은 다음과 같다.

```yaml
app:
  hold:
    seat-queue:
      enabled: ${HOLD_SEAT_QUEUE_ENABLED:true}
      max-per-seat: ${HOLD_SEAT_QUEUE_MAX_PER_SEAT:100}
      ttl: ${HOLD_SEAT_QUEUE_TTL:30s}
```

현재 구현은 "고유 사용자 100명"이 아니라 "좌석별 인증 요청 슬롯 100개" 기준이다. 이번 부하 테스트도 하나의 사용자 토큰을 재사용해 완전동시 요청 수를 검증했다.

### 3.2 Redis pre-lock

Redis pre-lock은 DB lock에 진입하기 전에 Redis에서 먼저 좌석 선점 가능성을 확인한다.

pre-lock 획득 실패 요청은 DB까지 가지 않고 빠르게 `409 Conflict`로 종료된다. 정상 hold 생성이 commit되면 Redis pre-lock은 hold 정보로 확정되고, 트랜잭션 실패 시에는 release된다.

### 3.3 큐 제한 + pre-lock 조합

`codex/redis-prelock-seat-queue-limit` 브랜치의 처리 순서는 다음과 같다.

1. 요청 좌석 목록 정규화
2. 좌석별 큐 티켓 획득
3. Redis pre-lock 획득
4. DB lock 및 도메인 상태 검증
5. Hold 저장
6. 트랜잭션 commit 후 pre-lock 확정
7. 요청 종료 시 큐 티켓 반환

이 구조의 목적은 DB까지 들어갈 가능성이 낮은 요청을 최대한 앞단에서 빠르게 탈락시키는 것이다.

## 4. 반환 정책

현재 좌석별 큐 초과는 `409 Conflict`로 반환한다.

| 상황 | 현재 상태 코드 | 의미 |
| --- | ---: | --- |
| 좌석별 큐 초과 | 409 | 해당 좌석 선점 경쟁에서 빠르게 탈락 |
| Redis pre-lock 실패 | 409 | 이미 선점 경쟁에서 밀림 |
| DB 상태상 좌석 사용 불가 | 409 | 이미 hold/reserved/sold 상태 |
| Redis 장애 | 503 | 좌석 선점 보호 계층 사용 불가 |

UX 관점에서는 `409`가 "이미 선점됨"에 가깝고, 운영 지표 관점에서는 좌석별 제한 초과를 `429 Too Many Requests`로 분리하는 선택지도 있다.

- backend
- postgres
- redis
- prometheus
- grafana
- postgres-exporter
- redis-exporter

각 케이스마다 다음 순서로 진행했다.

1. 대상 브랜치 checkout
2. `docker compose down -v --remove-orphans`
3. `docker compose up -d --build`
4. backend health 확인
5. `http://127.0.0.1:18080` 경유로 부하 테스트

부하 테스트 조건은 다음과 같다.

- 대상: 같은 show의 같은 좌석 1개

- 방식: gate를 사용한 완전동시 요청
- 토큰: 하나의 사용자 토큰 재사용

## 8. 분석

### 8.1 main

정합성은 유지된다. `200`은 항상 1건이다.

하지만 DB lock 경합이 크다. 동시 요청이 커질수록 평균 응답과 p95가 급격히 증가하고, 5000명부터 `500`이 관측됐다.

### 8.2 좌석별 큐 제한만 사용

폭주를 앞에서 제한한다는 방향은 맞지만 단독으로는 효과가 좋지 않았다.

큐를 통과한 요청들이 여전히 DB lock 경합으로 몰리고, Redis 큐 체크 비용이 추가된다. 이번 테스트에서는 main보다 느린 결과가 나왔다.

### 8.3 Redis pre-lock만 사용

500~3000명 구간에서 가장 큰 성능 개선을 보였다.

DB 진입 전에 Redis에서 대부분의 실패 요청을 빠르게 탈락시키기 때문에 DB lock 경합이 크게 줄어든다. 다만 5000명 구간에서는 네트워크 오류가 크게 발생했다. 이 부분은 애플리케이션 정책만의 문제로 단정하기 어렵고, 로컬 부하 생성기와 Docker/nginx 한계가 섞였을 가능성이 높다.

### 8.4 좌석별 큐 제한 + Redis pre-lock

가장 합리적인 운영 후보이다.

큐 제한은 순간 폭주를 앞단에서 누르고, pre-lock은 DB 진입 전에 성공 가능성이 낮은 요청을 빠르게 탈락시킨다. 이번 테스트에서 5000명까지 오류 없이 처리한 유일한 방식이다.

## 9. 남은 확인 과제

현재 결과만으로는 모든 `409`의 내부 원인을 분리할 수 없다. 큐 초과, pre-lock 실패, DB 상태상 좌석 사용 불가가 모두 `409`로 집계된다.

다음 단계에서는 응답 error code 또는 Prometheus metric을 추가하는 것이 좋다.

예시 metric:

- `hold.queue.entered`
- `hold.queue.rejected`
- `hold.prelock.acquired`
- `hold.prelock.rejected`
- `hold.db.lock.entered`
- `hold.db.conflict`
- `hold.created`

이 값을 추가하면 5000명 중 몇 명이 큐에서 탈락했고, 몇 명이 pre-lock에서 탈락했고, 몇 명이 DB까지 도달했는지 명확히 판단할 수 있다.

또한 5000명 이상에서 발생한 네트워크 오류는 다음 순서로 추적하는 것이 좋다.

1. frontend nginx access/error log
2. backend Tomcat thread/connection 로그
3. Docker Desktop CPU/Memory/Network 사용량
4. PostgreSQL connection pool 및 lock wait
5. Redis command latency
6. Windows socket/ephemeral port 상태
7. 부하 생성기 `.NET HttpClient` exception 세부 원인

## 10. 의사결정 메모와 추가 대안

### 10.1 5000건 동시요청 기준

이번 의사결정에서 기준으로 삼은 값은 `한 좌석에 5000개 요청이 거의 동시에 몰리는 burst 상황`이다.

여기서 `1ms 이내 5000건`이라는 표현은 조심해야 한다. 문자 그대로 해석하면 초당 500만 요청에 해당하고, 단일 로컬 PC, Docker Desktop, 단일 WAS 인스턴스의 처리량 목표로 보기 어렵다. 이번 테스트가 확인한 것은 5000개 클라이언트 작업을 동시에 출발시켰을 때 애플리케이션이 정합성을 유지하고 실패 요청을 명확하게 반환하는지이다.

운영 목표는 아래처럼 잡는 편이 더 정확하다.

```text
같은 좌석에 5000개 burst 요청이 몰렸을 때
1건만 Hold 생성에 성공하고,
나머지는 5xx나 네트워크 오류가 아니라 빠른 실패 응답으로 정리된다.
```

이 기준에서는 `좌석별 큐 제한 + Redis pre-lock`이 가장 좋은 결과를 보였다. 특히 5000명 테스트에서 `200` 1건, `409` 4999건, 네트워크 오류 0건, `500` 오류 0건으로 끝났다.

### 10.2 채택 판단

이 방식은 모든 요청을 공정하게 오래 기다리게 만드는 방식이 아니라, 성공 가능성이 낮은 요청을 빠르게 탈락시키는 방식이다. 티켓팅 UX에서는 이미 한 좌석에 수천 개 요청이 들어온 시점부터 후발 요청의 성공 가능성이 극히 낮기 때문에, DB까지 끌고 가서 느리게 실패시키는 것보다 빠르게 실패를 반환하는 편이 더 낫다.

개발자 관점에서도 이 방식은 병목 위치를 분명하게 만든다.

- queue는 좌석별 순간 진입량을 제한한다.
- pre-lock은 DB 진입 전 중복 선점 시도를 차단한다.
- DB lock과 `@Version`은 최종 정합성 보호선으로 남긴다.

따라서 현재 단계에서는 이 방식을 채택한다.

### 10.3 추가 대안

채택 방식과 별개로, 트래픽 규모가 더 커질수록 아래 대안을 함께 검토해야 한다.

| 대안 | 장점 | 주의점 |
| --- | --- | --- |
| WAS scale-out | 전체 HTTP 처리량과 조회/인증 API 처리 여유 증가 | 동일 좌석 정합성은 Redis/DB 공유 보호선이 계속 필요 |
| Redis Cluster 또는 admission 전용 Redis | queue/pre-lock 부하 분산 | Lua script key hash tag와 장애 복구 정책 필요 |
| Nginx/API Gateway rate limit | Tomcat 전에 폭주 차단 | 좌석 ID 단위 세밀한 정책은 구현이 까다로움 |
| Waiting Room | 오픈 직후 유입량을 서비스 처리량에 맞춰 평탄화 | 순번 공정성, 이탈 UX, 재접속 정책 필요 |
| 좌석별 빠른 실패 UX | 이미 경쟁이 끝난 좌석에 대해 사용자가 오래 기다리지 않음 | 실패 메시지가 납득 가능해야 함 |
| SSE/WebSocket availability push | 이미 선점된 좌석 클릭 자체를 줄임 | fan-out 비용과 모바일 환경 고려 필요 |
| show/event 단위 파티셔닝 | 대형 이벤트 간 hot spot 분리 | 운영 복잡도와 데이터 라우팅 설계 필요 |

### 10.4 UX 제안

좌석 클릭 직후 모든 사용자를 같은 화면에서 오래 기다리게 두는 것보다, 성공 가능성이 낮은 요청은 빠르게 실패시키고 다음 행동을 제안하는 편이 좋다.

추천 UX는 다음과 같다.

1. queue/pre-lock 실패 시 "이미 다른 사용자가 선점 중입니다"를 즉시 표시한다.
2. 같은 구역의 다른 좌석을 바로 추천한다.
3. 좌석 상태를 짧은 주기로 갱신하거나 SSE/WebSocket으로 업데이트한다.
4. 이벤트 오픈 직후 전체 유입이 큰 경우에는 좌석 선택 화면 앞에 waiting room을 둔다.
5. 실패 요청이 반복되는 사용자는 짧은 cool-down을 적용해 서버와 사용자 모두의 피로도를 낮춘다.

## 11. 최종 판단

운영 후보로는 `좌석별 큐 제한 + Redis pre-lock`을 추천한다.

티켓팅 도메인에서는 한 좌석에 이미 많은 요청이 들어온 시점부터 후발 요청의 성공 가능성이 극히 낮다. 따라서 모든 요청을 DB까지 보내 느리게 실패시키는 것보다, 좌석별 큐와 pre-lock에서 빠르게 실패를 반환하는 편이 개발자 관점과 UX 관점 모두에서 더 합리적이다.

다만 실제 서비스 정책으로 확정하려면 다음 결정이 필요하다.

1. 좌석별 큐 제한을 "요청 슬롯 수"로 볼지, "고유 사용자 수"로 볼지 결정한다.
2. 큐 초과 응답을 `409`로 유지할지 `429`로 분리할지 결정한다.
3. 10000명 이상 검증은 단일 로컬 PC가 아니라 분산 부하 생성 환경에서 다시 측정한다.
